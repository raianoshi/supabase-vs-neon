Team Git & GitHub Workflow Guide
for First-Time Next.js Collaboration

1. Overview

This guide explains the standard Git and GitHub workflow for a team collaborating on a Next.js project for the first time. It covers repository setup, branch protection, feature branching, pull requests, and the commands your team will use daily.

2. Initial Project Setup (One Person Only)

2.1 Create the GitHub Repository

One team member creates a new GitHub repository. If the Next.js project already exists locally, do not initialize the repo with a README or .gitignore — GitHub will warn about conflicts.

2.2 Create the Next.js Project Locally

npx create-next-app@latest my-project
cd my-project

2.3 Initialize Git

git init
git add .
git commit -m "Initial Next.js project setup"

2.4 Connect Local Repo to GitHub

git remote add origin https://github.com/<org>/<repo>.git
git branch -M main
git push -u origin main

3. Protecting the main Branch

Protecting the main branch ensures that no one accidentally breaks the production-ready code.

3.1 Recommended Protection Rules

  • Require pull requests before merging
  • Require at least one reviewer
  • Disallow direct pushes to main
  • Require branches to be up-to-date before merging

These settings are found under:
GitHub → Repository → Settings → Branches → Add Rule

4. Team Setup: Cloning the Repository

Each team member clones the repository and installs dependencies:

git clone https://github.com/<org>/<repo>.git
cd repo
npm install

5. Standard Team Workflow: Feature Branching

Every new task or feature should be developed in its own branch. This keeps work isolated and makes reviewing changes easier.

5.1 Create a New Branch

git checkout -b feature/add-login-page

5.2 Work on the Feature

Run the development server:

npm run dev

5.3 Stage and Commit Changes

git add .
git commit -m "Add login page UI"

5.4 Push the Branch to GitHub

git push -u origin feature/add-login-page

6. Pull Requests (PRs)

Pull requests allow the team to review code before merging it into main.

6.1 Steps to Create a PR

  1. Go to the repository on GitHub
  2. Click Compare & Pull Request
  3. Ensure the base branch is main
  4. Add a clear title and description
  5. Request reviewers
  6. Address feedback and update the branch if needed
  7. Merge when approved
After merging, delete the feature branch on GitHub to keep the repo clean.

7. Keeping Your Local Repo Updated

7.1 Update main

git checkout main
git pull origin main

7.2 Sync a Feature Branch with Latest main

git checkout feature/add-login-page
git pull origin main

If merge conflicts occur, resolve them manually, then:

git add .
git commit

8. Cleaning Up Local Branches

git branch -d feature/add-login-page
git fetch --prune

9. Common Git Commands Reference

Status

git status

List Branches

git branch

Switch Branch

git checkout <branch>

Create + Switch

git checkout -b <branch>

Stage All Changes

git add .

Commit

git commit -m "Message"

Push

git push

Pull

git pull

10. Recommended Team Conventions

Branch Naming

  • feature/<name>
  • bugfix/<name>
  • hotfix/<name>
  • chore/<name>

Commit Message Style

  • Use clear, present-tense descriptions
  • Examples:
    • Add login form validation
    • Fix navbar layout

Pull Request Best Practices

  • Keep PRs small and focused
  • Include screenshots for UI changes
  • Link related issues

Environment Variables

Never commit .env files. Use .env.local and ensure .env* is in .gitignore.

11. Summary

By following this workflow, your team will maintain a clean, stable codebase while collaborating efficiently on your Next.js project. The combination of protected branches, feature branching, and pull requests ensures quality and consistency across the entire project.
